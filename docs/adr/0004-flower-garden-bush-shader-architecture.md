# Flower-garden bush-shader architecture

The flower-garden theme renders rose cells (the existing `roseBudDeform.sksl` + `CellRoseBud`) but had no scenery — no connection between the roses and the ground, and no foliage. We added four assets: `stem.png` (a tall thin stem texture), `rose_base.png` (the calyx), and `leaf1–4.png`. Roses float disconnected, so the table reads as a grid of disembodied flowers.

We grouped stems into **bushes** (3–5 stems each, 4–6 bushes for the 19-rose `spanishPresentTable2Plural` table, derived from rose count) and draw all stems, calyces, and leaves of one bush inside a single SkSL shader pass. The shader is one fullscreen `Rect` per bush; the uniforms carry an array of stem configs and an array of leaf configs. The shader runs three draw phases per bush in this order: stems (including the calyx at the rose's current position), inner-side leaves, outer-side leaves. This gives correct per-leaf z-ordering (some leaves hidden by the stem band, some in front) without a multi-pass per-bush render. Six image samplers (1 stem + 1 calyx + 4 leaf variants) stay well within SkSL's limit.

The shader shape is a quadratic bezier from bush-base to rose-top with a control point on the *outer* side of the base→top line (away from the bush center), so stems fan outward. Width is a linear taper (~3–5 px at ground, ~15–20 px at calyx). The calyx moves with the rose — the shader pulls `(x, y, scale)` from the word-sprite bridge's `layoutX`/`layoutY`/`layoutScale` shared values, so rose drag and motion loop propagate to the calyx and (via parallax) to the leaves with no JS roundtrip. Leaf motion uses a per-leaf parallax weight tied to its position along the stem (low = barely moves, high = moves with the rose). The bush config (count, assignment, base point, control points, leaf positions, leaf `side` pre-computation) is generated once from a `mulberry32` seeded by `table.id`, so the same table re-renders identically.

**Considered options:** one shader per stem (simpler, but each stem's leaves can only be uniformly behind or in front of that stem — wrong look); two-pass (stems then all leaves on top, no per-leaf z — leaves on the far side of the stem still appear in front, visually wrong); three-pass with pre-computed `side` but a separate shader per layer (same correctness, but triple the Skia compile time and three fullscreen rects per bush instead of one). The one-shader-per-bush design is the smallest shape that gets z-correct rendering within a bush.

**Trade-offs:** the per-bush shader's uniform array scales with `MAX_STEMS_PER_BUSH × MAX_LEAVES_PER_STEM` (5 × 9 = 45 leaves, ~305 floats per bush uniform); going higher costs Skia compile time. Bushes never overlap, so cross-bush z-ordering is irrelevant — the bushes themselves draw bottom-to-top by base-Y to give a coherent ground line.

**Cross-references:** sits inside the `Scenery` slot of the `Theme` interface (ADR-0003); no change to the rose-layer (`FlowerGardenWordSpriteTableLayer`) or the word-sprite bridge contract. New terms in `CONTEXT.md`: `Bush`, `Bush base`, `Calyx`, `Rose stem`, `Rose leaf`, `Leaf side`.
