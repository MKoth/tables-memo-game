# Flower-garden butterfly Roamer

The flower-garden theme had no Roamer realisation (the slot rendered an empty `View`). We added butterflies to fill it: a butterfly is a per-instance runtime that owns body+wing image refs, position, angle, and flight state. The roamer sim is per-theme (no shared abstractions across the koi and butterfly paths — the `Roamer` contract is at the `Theme` level, not at the sim level). The 9-variant wing-pair set is assembled inside a single SkSL shader (one `Rect` per butterfly), with body and the two wings composited by per-region UV offsets. Wing flapping is a per-side UV-thin/thick stretch driven by `sin(wingPhase)`, and the *difference* between the left and right wing phases is the per-frame turn rate — that asymmetry replaces the koi's `wanderAngle + boundary` model and is what gives butterflies their erratic steering. The koi sim keeps its current shape; the butterfly sim is a new, separate, theme-owned path.

**Field flowers** are scenery that the roamer sim can read and write. We model them as scenery records with one extra field — `occupant: roamerIndex | null` — that names the sitting butterfly, or `null` if free. Roamers never write to any other part of scenery; the occupant slot is the only cross-context data path between Scenery and Roamer. The roamer sim reads the slot to decide whether to land, wait, or pick a new flower, and writes it on land and on lift-off. Field flowers live in the roamer zone at a fixed count (12), with placement seeded for deterministic per-session layout. They are not WordSprites — they never appear in the WordSprite zone and never carry words; the match field's 1:1 invariant is preserved independently.

**Sitting draw pass.** The koi has a flicker problem when a fish moves between the pool and the bubble (it momentarily re-anchors at the wrong world coord). The butterfly's "sitting on a flower" is a structurally similar problem: the roamer needs to render in two visually distinct ways (wings-up at full size in flight, wings-folded smaller on a flower) while staying at the same world position. We solve it by giving each roamer two render passes that both read the same runtime and the same shared `positions` array; the passes are gated by the runtime's `flight state`. State is in `WAIT_AT_TAKEN_FLOWER`, `SITTING`, or `LIFTING_OFF` → sitting pass draws (with the sitting scale and on-flower offset). State is in any of the four `FLYING_*` or `APPROACH_FLOWER` → flying pass draws. The transition is a one-frame toggle with no coordinate hand-off, so the two passes never draw the same butterfly in the same frame and the shared `positions` write on the state transition is what keeps the roamer at the flower anchor when the flying pass takes over again.

**State machine** is a flat list of seven states, one active at a time per roamer: `FLYING_IDLE` (flapping in place, small drift on all axes, no target), `FLYING_CRUISE` (wander target + wing-phase steering toward it), `FLYING_TURN` (deliberate heading change to a new wander target — often elided into CRUISE for small heading deltas), `APPROACH_FLOWER` (locked path to a specific flower anchor, wing phase still steers), `WAIT_AT_TAKEN_FLOWER` (position holds near the anchor, polls `occupant` for `null`), `SITTING` (on anchor, arc micro-motion and on-place turning, legs visible), `LIFTING_OFF` (scale lerps from sitting to full at the anchor, then transitions to `FLYING_CRUISE`). Wander-target pick is the 20% gate: 20% chance the next target is a field flower (state → `APPROACH_FLOWER`), 80% a new wander point (state stays in `FLYING_CRUISE`).

**Wing pair spawn policy** is "shuffle 9, deal, then round-robin from random offset": shuffle the 9 pair indices, deal one each to the first 9 butterflies in spawn order; for any overflow the i-th extra butterfly uses pair `(startOffset + i) % 9` where `startOffset` is a random integer in `0..8` rolled once per session. Wings never change after spawn — no molting, no lift-off reroll. The match field's 8-roamer count fits inside the 9 variants, so duplicates are only an issue in the decorative roamer layer (which uses the same policy independently).

**Considered options:**
- *Reuse the koi sim and just swap visuals.* Tempting because the koi sim already exists, but the wing-as-steering model is incompatible with `wanderAngle + boundary`. Either we'd have to fake the steering in the shader (cosmetic only — feels koi-shaped), or refactor the koi sim to support both, which couples the two themes' motion models. Per-theme sims keep them independent.
- *Field flowers as WordSprites.* Would unify "anything a roamer can land on" but pulls field flowers into the WordSprite zone layout, the word-sprite bridge, and the entrance/exit animation system. They're scenery, not words; the separation is cheaper.
- *Field flowers as pseudo-roamers in the roamer pool.* Would reuse the existing spatial-hash and target machinery, but pseudo-roamers would need a new "no movement" mode, an `occupant` field, and a different draw path. The slot-on-scenery model is smaller.
- *Two separate roamer runtimes (flying + sitting) per butterfly.* Position hand-off between runtimes on lift-off is the very thing that flickers in the koi→bubble case. One runtime, two draw passes, is the same pattern that worked for the koi fix.
- *Event-based lift-off notifications for `WAIT_AT_TAKEN_FLOWER`.* Cleaner conceptually, but adds an event bus for what is per-frame polling against a 12-entry array.
- *Allow wing swap on lift-off.* Re-uses the uniqueness pool as butterflies cycle. The 9-pair pool is small enough that this can visibly produce "the same butterfly" reappearing — surprising without context. The shuffle-then-fixed policy is predictable.
- *6-leg single global phase.* Half the shader cost, but reads as one leg with 6 copies animating in unison, not a walking insect.

**Consequences:** the roamer sim now has two divergent code paths (koi, butterfly), each in its own folder under `themes/<theme>/roamer/`. The shared `Roamer` Theme contract (ADR-0003) is unchanged — both paths conform to it — but there is no shared runtime helper or shared update function. If a third roaming theme is added later, the pattern "one folder per roamer realisation" is the one to follow; do not extract a common sim.

The `Field flower` model introduces a one-way write from Roamer to Scenery (occupant slot). The Scenery side must tolerate the occupant field being written by an external system; it does not own that field's lifecycle beyond the visual draw. Scenery reads `occupant` only if it wants to render a "taken" indicator (out of scope for v1).

**Cross-references:** sits inside the `roamer` slot of the `Theme` interface (ADR-0003); new terms in `CONTEXT.md`: `Butterfly`, `Wing pair`, `Field flower`, `Flower anchor`, `Occupant`, `Wing phase`, `Flight state`, `Sitting draw pass`, `Leg phase`. The layer-flip draw pattern is the same pattern that resolved the koi pool→bubble flicker in `releaseCapturedFishWorklet` (`themes/undersea/roamer/capture/releaseFishToPool.ts`); no code is shared between the two paths.
